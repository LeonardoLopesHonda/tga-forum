# Supabase Migrations

Schema is owned by the migration files in this directory. `supabase start` auto-applies them locally; `supabase db push --linked` applies them to the linked remote project.

The backend no longer calls `Base.metadata.create_all()` on boot. The SQLite test path still uses `create_all` against an in-memory engine — that's a separate codepath and unaffected by this directory.

## Workflow

Adding a migration:

```
supabase migration new <roadmap_row_slug>     # creates <timestamp>_<slug>.sql
# write the SQL
supabase db reset                             # rebuilds local DB from scratch, confirms the migration applies cleanly
supabase db push --linked                     # applies to prod
```

## Conventions

- **One migration per `ROADMAP.md` row.** Filename slug mirrors the roadmap row name. If a change doesn't fit any row, the roadmap is missing a row — add it first.
- **Additive on existing tables.** New columns are nullable or have a default. Never `DROP TABLE` or otherwise destroy user data. Backfill explicitly in SQL when an existing row needs a value for a new column.
- **Views must be recreated when their source columns change.** `post_with_username` and `comment_with_username` are real views. A migration that adds a column to `post` or `comment` and wants that column visible through the view must `DROP VIEW ... CASCADE; CREATE VIEW ...` in the same migration. See `<timestamp>_initial.sql` for the canonical view definitions.

## Baseline record

The initial migration (`<timestamp>_initial.sql`) was generated from the live prod schema via:

```
supabase db dump --schema public --linked -f supabase/migrations/<timestamp>_initial.sql
```

It was applied to prod via `supabase migration repair --status applied <timestamp> --linked`, not `db push` — the schema it describes was already in prod at the time of capture. Do not edit this file; future schema changes are new migrations.

`supabase db diff --linked` should return empty against the linked prod project, with one known exception: `drop extension if exists "pg_net"`. `pg_net` is Supabase-managed and enabled on the hosted project but not locally; the app doesn't use it, so the drift is accepted. Any other non-empty diff means something in prod isn't captured here and needs reconciliation before any new migration is pushed.
